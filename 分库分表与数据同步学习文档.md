# 分库分表与数据同步学习文档

## 一、目标
- 业务库：ShardingSphere-JDBC 分库分表写入 MySQL。
- 查询策略：按 `userId` 走 MySQL 单分片查询；全量分页走 Elasticsearch。
- 同步链路：MySQL -> Debezium Server -> Kafka -> Kafka Connect Elasticsearch Sink -> Elasticsearch。

## 二、分库分表实现
- 逻辑表：`t_order`。
- 物理节点：`ds_0.t_order_0`、`ds_0.t_order_1`、`ds_1.t_order_0`、`ds_1.t_order_1`。
- 分库算法：`ds_${user_id % 2}`。
- 分表算法：`t_order_${user_id < 1000000000 ? 0 : 1}`。
- 主键策略：SNOWFLAKE。

## 三、数据同步实现
### 1) Debezium Server（MySQL Source）
关键配置：
- `debezium.source.connector.class=io.debezium.connector.mysql.MySqlConnector`
- `debezium.source.database.connectionTimeZone=UTC`
- `debezium.source.decimal.handling.mode=double`
- `debezium.transforms=unwrap,route`
- `debezium.transforms.unwrap.type=io.debezium.transforms.ExtractNewRecordState`
- `debezium.transforms.route.replacement=order`

效果：
- 将 CDC 事件展开后写入 Kafka `order` topic。
- `amount` 输出为数值，避免 Decimal bytes 导致 ES 落库失败。

### 2) Kafka Connect Elasticsearch Sink
关键配置：
- 只消费 `topics=order`
- `transforms=extractPayload,renameFields,createKey,extractKey`
- `extractPayload`：取 Debezium 结构中的 `payload`
- `renameFields`：`user_id:userId,create_time:createTime`
- `createKey + extractKey`：用 `id` 作为 ES 文档 `_id`
- `behavior.on.null.values=ignore`
- `behavior.on.malformed.documents=ignore`

效果：
- ES 文档结构与后端字段一致。
- Upsert 以 `id` 幂等更新。

## 四、本次排障结论
1. `Key is used as document id and can not be null`
- 原因：部分消息 key 为空或未正确提取 `id`。
- 处理：`extractPayload -> createKey(id) -> extractKey`。

2. ES sink `illegal document`
- 原因：`amount` 为 Debezium Decimal bytes，ES 映射不兼容。
- 处理：`debezium.source.decimal.handling.mode=double`。

3. 时间偏移 8 小时
- 现象：ES 时间与预期上海时间换算不一致。
- 根因：应用 JDBC 与数据库时区配置不一致导致写库时已偏移。
- 处理：
  - ShardingSphere JDBC URL 改为 `serverTimezone=Asia/Shanghai`
  - Debezium `database.connectionTimeZone=UTC`
- 验证口径：
  - API 展示为上海时间字符串。
  - Kafka/ES 存储为 UTC (`...Z`)。
  - 同一订单以 epoch milli 对比应一致。

## 五、验证步骤
1. 新增订单：`POST /api/orders`
2. 查 MySQL：按 `id` 查询 `create_time`
3. 查 Kafka：消费 `order` topic 最新消息，确认 `create_time`
4. 查 ES：`GET /order/_search` 按 `id` 验证文档
5. 查接口：`GET /api/orders` 确认前端展示时间

## 六、稳定运行要点
- 修改 Debezium 配置后必须重启容器，配置不会热加载。
- 连接器失败先看 `status` 再看 `kafka-connect` 日志。
- 主题历史脏数据会影响回放，必要时重建 topic 并重导。
- 统一时间口径：存储 UTC，展示按 Asia/Shanghai。
