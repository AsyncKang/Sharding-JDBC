# 分库分表与数据同步学习文档

## 1. 学习目标与范围

本文目标是让你从“能跑通”进阶到“能讲清楚为什么这么设计、每一段链路底层在做什么”。

覆盖链路：
- 写入链路：前端 -> Spring Boot -> MyBatis -> ShardingSphere-JDBC -> MySQL
- 同步链路：MySQL binlog -> Debezium -> Kafka -> Kafka Connect -> Elasticsearch
- 查询链路：MySQL 单分片查询 + Elasticsearch 全量/条件查询

---

## 2. 先建立全局心智模型

同一条订单数据会经历两条路径：

1) 事务路径（强一致）
- 应用写入 MySQL，事务提交后数据立即可查（按分片键场景）。

2) 检索路径（最终一致）
- MySQL 变更通过 binlog 进入 CDC 链路，最终落到 Elasticsearch。
- 这条链路有传输和消费延迟，所以“写入 MySQL 立即查 ES”可能短暂查不到。

你可以把它理解为：
- MySQL 是事实源（Source of Truth）
- ES 是检索副本（Read Model）

---

## 3. 组件一：ShardingSphere-JDBC（写入与路由）

### 3.1 它在链路中的职责
- 接管 JDBC 层，按分片规则改写 SQL 并路由到具体库表。
- 让业务代码仍面向逻辑表 `t_order` 开发，不直接拼物理表名。

### 3.2 本项目配置位置
- `backend/src/main/resources/sharding.yaml`

### 3.3 关键知识点

#### 知识点 A：逻辑表与物理表
- 逻辑表：代码中写的 `t_order`
- 物理表：真实存储的 `ds_0.t_order_0`、`ds_0.t_order_1`、`ds_1.t_order_0`、`ds_1.t_order_1`

对应配置：
- `rules[0].tables.t_order.actualDataNodes=ds_${0..1}.t_order_${0..1}`

#### 知识点 B：分库算法
对应配置：
- `shardingAlgorithms.order_database_inline.algorithm-expression: ds_${user_id % 2}`

含义：
- `user_id` 对 2 取模，决定进 `ds_0` 还是 `ds_1`。

#### 知识点 C：分表算法
对应配置：
- `shardingAlgorithms.t_order_table_inline.algorithm-expression: t_order_${user_id < 1000000000 ? 0 : 1}`

含义：
- `user_id` 小于阈值进 `t_order_0`，否则进 `t_order_1`。

#### 知识点 D：主键生成
对应配置：
- `keyGenerators.snowflake`

含义：
- 分布式生成 Long ID，避免多库场景下自增主键冲突。

注意：
- Long ID 到前端必须按字符串传输，否则 JS 精度会丢失。

### 3.4 代码关联
- 写入入口：`backend/src/main/java/com/example/shardingdemo/service/OrderService.java` 的 `create()`
- SQL 映射：`backend/src/main/resources/mapper/OrderMapper.xml`

## 4. 组件二：MySQL 与 binlog（CDC 数据源）

### 4.1 它在链路中的职责
- 保存事务数据。
- 在事务提交时，把变更写入 binlog，供 CDC 系统订阅。

### 4.2 什么是 binlog
- binlog 是 MySQL 的二进制日志，记录数据库变更事件。
- Debezium 读取的不是表文件，而是 binlog 事件流。

### 4.3 “MySQL 写入成功后，binlog 如何记录”的细化说明

以一条 `INSERT INTO t_order ...` 为例：
1. 事务执行期间，变更先在 InnoDB 内部处理。
2. 提交阶段，MySQL 将变更写入 binlog（常见推荐格式是 ROW）。
3. binlog 事件包含：
   - 表标识（哪个库、哪张表）
   - 操作类型（INSERT/UPDATE/DELETE）
   - 行级数据（变更前/后）
   - 位点信息（file + position 或 GTID）
4. 事务真正提交成功后，这个 binlog 事件才可被下游读取。

关键点：
- binlog 是“提交后变更”的事件源，不会采集未提交事务。

---

## 5. 组件三：Debezium Server（把 binlog 变成 Kafka 消息）

### 5.1 它在链路中的职责
- 作为 CDC 采集器，持续消费 MySQL binlog。
- 把 binlog 事件转换为结构化消息并发送到 Kafka。

### 5.2 本项目配置位置
- `debezium/server/conf/application.properties`

### 5.3 “Debezium 如何读取 binlog”的细化说明

Debezium 读取流程（核心机制）：
1. 使用 MySQL 复制协议连接数据库（行为类似一个 replica）。
2. 先根据位点（offset）确定从哪里开始读。
3. 持续接收 binlog 事件流。
4. 根据 schema history 还原该时刻表结构，正确解析字段类型。
5. 把事件转换为 Connect 记录（key/value）。
6. 应用 SMT（如 `unwrap`、`route`）后发往 Kafka。

### 5.4 关键配置与意义

#### 连接与来源
- `debezium.source.connector.class=io.debezium.connector.mysql.MySqlConnector`
- `debezium.source.database.hostname/port/user/password`

作用：
- 定义 CDC 数据源与连接方式。

#### 监听范围
- `debezium.source.database.include.list`
- `debezium.source.table.include.list`

作用：
- 限定只采集订单相关库表，减少无关事件。

#### 时区与数值
- `debezium.source.database.connectionTimeZone=UTC`
- `debezium.source.decimal.handling.mode=double`

作用：
- 统一时间解释，避免 8 小时偏移。
- Decimal 转数值，避免 sink 落库冲突。

#### 事件变换（SMT）
- `debezium.transforms=unwrap,route`
- `unwrap`：提取业务字段，去掉 Debezium envelope。
- `route`：把多物理表 topic 收敛为 `order`。

### 5.5 Debezium 状态文件为什么重要

- `debezium/server/data/offsets.dat`
  - 保存消费位点；决定重启后从哪里继续。
- `debezium/server/data/schemahistory.dat`
  - 保存 DDL 历史；决定字段解析是否正确。

缺失影响：
- offset 丢失可能导致重复回放。
- schema history 丢失可能导致解析失败或字段错位。

---

## 6. 组件四：Kafka（变更事件总线）

### 6.1 它在链路中的职责
- 承接 Debezium 生产的 CDC 消息。
- 让上游采集与下游消费解耦。

### 6.2 本项目中的作用
- 统一主题：`order`
- 上游生产者：Debezium Server
- 下游消费者：Kafka Connect ES Sink

### 6.3 学习重点
- Topic：消息逻辑分类。
- Partition：并行与有序的基本单位。
- Offset：消费位点，决定“消费到哪里了”。

---

## 7. 组件五：Kafka Connect + ES Sink（把消息写入 ES）

### 7.1 它在链路中的职责
- 订阅 Kafka `order` 消息。
- 做字段变换、主键提取、写入策略控制。
- 将数据落地到 Elasticsearch 索引。

### 7.2 本项目配置位置
- `debezium/connect/es-sink-order.json`

### 7.3 关键配置与意义

#### 订阅来源
- `topics=order`

意义：
- 只消费业务统一主题，避免多 topic 管理复杂度。

#### 文档主键与幂等
- `key.ignore=false`
- `createKey + extractKey` 从 value 取 `id` 作为 key
- `write.method=upsert`

意义：
- ES 文档 `_id=id`，重复事件会覆盖更新而不是新增重复文档。

#### 字段结构对齐
- `extractPayload`
- `renameFields: user_id:userId, create_time:createTime`

意义：
- 让 Kafka 消息结构与 ES/后端字段命名一致。

#### 异常处理
- `behavior.on.null.values=ignore`
- `behavior.on.malformed.documents=ignore`

意义：
- 避免 tombstone 或脏记录导致任务退出。

---

## 8. 组件六：Elasticsearch（检索副本）

### 8.1 它在链路中的职责
- 接收订单副本数据。
- 承接全量分页和条件检索。

### 8.2 本项目中的使用方式
- 索引：`order`
- 全量分页：`findAll` + 排序
- 条件检索：关键词、userId、时间范围组合

### 8.3 为什么查询分流
- `userId` 查询在 MySQL 可精确路由到单分片，成本低。
- 全量/模糊/多条件检索在 ES 更高效。

---

## 9. 组件七：后端查询层（MySQL 与 ES 分流）

### 9.1 MySQL 路径
- `OrderService.pageByUser()`
- 通过 `OrderMapper.selectByUserId()` 查询。

特点：
- 命中单分片，响应稳定。

### 9.2 ES 路径
- `OrderEsQueryService.pageAll()`
- `OrderEsQueryService.search()`

特点：
- 支持全量分页与组合条件检索。

## 11. 典型问题与定位方法

### 11.1 ES 没数据
排查顺序：
1. MySQL 是否有新数据。
2. Kafka `order` topic 是否有新消息。
3. Connect `status` 是否 RUNNING。
4. Connect 日志是否有 `malformed document`。

### 11.2 时间不一致（差 8 小时）
排查点：
- JDBC `serverTimezone`
- Debezium `database.connectionTimeZone`
- MySQL `@@time_zone` / `@@system_time_zone`

### 11.3 ID 显示不准确
- 原因：JS Number 精度限制。
- 处理：后端 Long 序列化成字符串。

### 12.1 首次初始化
```bash
docker network create canal-net
```

```bash
docker run -d --name kafka --network canal-net -p 9092:9092 \
  -e KAFKA_NODE_ID=1 \
  -e KAFKA_PROCESS_ROLES=broker,controller \
  -e KAFKA_LISTENERS=PLAINTEXT://:9092,CONTROLLER://:9093 \
  -e KAFKA_ADVERTISED_LISTENERS=PLAINTEXT://kafka:9092 \
  -e KAFKA_CONTROLLER_LISTENER_NAMES=CONTROLLER \
  -e KAFKA_CONTROLLER_QUORUM_VOTERS=1@kafka:9093 \
  -e KAFKA_INTER_BROKER_LISTENER_NAME=PLAINTEXT \
  -e KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR=1 \
  -e KAFKA_TRANSACTION_STATE_LOG_REPLICATION_FACTOR=1 \
```
